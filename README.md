# SportsStats

Бэкенд для учёта хоккейной статистики с ролями «клиент» (просмотр) и «админ» (управление). Позволяет создавать турниры с настраиваемыми правилами (очки, овертаймы, заявки), вести команды и игроков, управлять матчами и фиксировать голы с детализацией (забивший, ассистенты, тип гола). На основе сыгранных матчей автоматически рассчитывается турнирная таблица.

**Архитектура:** Clean/DDD (домен → сценарии → инфраструктура → HTTP). REST API.

## Доменная модель

### Агрегаты и сущности

- **`Tournament`** — жизненный цикл: *Draft* → *Registration* → *InProgress* → *Finished*. Хранит набор id заявленных команд, опциональное фото. Старт/финиш и смена статуса сопровождаются **инвариантами** (например, нельзя начать без правил и минимум двух команд; нельзя завершить при незавершённых матчах).
- **`Match`** — домашняя/гостевая команды, расписание, статус (*Waiting* / *InProgress* / *Finished*), счёт, владельческие типы победы (`MatchWinType`: регей, овертайм, ничья и т.д.), копия правил турнира, составы (id игроков по командам), коллекция **`GoalEvent`**.
- **`Team`**, **`Player`** — команды и игроки; у игрока **позиция** (нападающие, защитники, вратарь), опциональные фото, гражданство, номер, дата рождения.
- **`TeamStats`** — турнирная таблица в разрезе *(команда + турнир)*: игры, победы в основное время/ОТ, ничьи, поражения в основное время/ОТ, очки; обновляется при завершении матча.
- **`GoalEvent`** — информация о голе (автор, ассистенты, ситуация)

### Правила турнира (`TournamentRules`)

Сериализуются в **JSON (jsonb)** в PostgreSQL. Включают три осмысленных блока:

- **`MatchTimeRules`** — периоды, овертайм, буллиты, допустимость ничьей и т.д.
- **`MatchRosterRules`** — минимальное/максимальное число игроков по позициям и в сумме (согласуется с проверкой состава в **`MatchService`** при старте матча).
- **`MatchPointsRules`** — сколько очков за победу в регейме, в ОТ, ничью и т.д. (связь с `MatchWinType` при начислении в **`MatchFinishService`**).

### Доменные сервисы

- **`MatchService`** — создание матча в допустимой фазе турнира, валидация заявленных на матч игроков при старте матча относительно `MatchRosterRules`.
- **`TournamentRankingService`** — сортировка списка `TeamStats` внутри одного турнира

### Общие абстракции

- **`BaseEntity`**, идентификаторы; **`IAggregateRoot`** у ключевых корней.
- **`ITimeProvider`** — абстракция «текущего времени» (реализация `SystemTimeProvider` в инфраструктуре), используется в сценариях старта/финиша.

Исключения бизнес-логики в основном как **`ArgumentException`** — глобальный обработчик отдаёт **400** с текстом, прочие ошибки — **500** (см. ниже).

## Слой Application

Крупные **сервисы сценариев** (скоуп *Scoped* в DI):

- **`TournamentApplicationService`** — CRUD-операции по турниру, смена статусов, установка правил, список/поиск, выборки «по дате с матчами», агрегаты для календаря/результатов по команде.
- **`TeamApplicationService`**, **`PlayerApplicationService`** — команды и игроки, пагинация, поиск, редактирование основной информации.
- **Матчи (разбивка по ответственности):**
  - **`MatchLifecycleService`** — создание матча в турнире, старт, смена основной информации;
  - **`MatchGoalService`** — голы и детализация гола;
  - **`MatchRosterService`** — заявка состава;
  - **`MatchFinishService`** — завершение и **актуализация `TeamStats`** по правилам очков;
  - **`MatchQueriesHandler`** — чтение/проекции в DTO (матч, расписания, фильтры по турниру/дате/команде).
- **`TeamStatsApplicationService`** — выдача турнирной/командной статистики.

DTO и мапперы лежат рядом по областям (турниры, матчи, игроки, статистика).

## HTTP API (контроллеры)

Базовый префикс: **`/api/...`**. Классические REST-подобные маршруты (ниже — смысловые группы, не полный перечень полей body).

| Контроллер | Маршрут | Назначение |
|------------|---------|------------|
| **TournamentsController** | `api/Tournaments` | Список/детали турнира, турниры по дате с матчами, создание, старт / регистрация / финиш, регистрация команд, установка правил, смена названия/фото |
| **TeamsController** | `api/Teams` | Список, по турниру, календарь/результаты команды, создание, смена основной информации |
| **PlayersController** | `api/Players` | Список, поиск, по команде, деталь, список позиций с описанием, создание, смена основной информации |
| **TournamentMatchesController** | `api/tournaments/{tournamentId}/matches` | Календарь и завершённые матчи турнира (пагинация), матчи на дату, создание матча |
| **MatchesController** | `api/Matches` | Подробная информация матча, голы, составы, старт/финиш, детализация гола, смена основной информации |
| **TeamStatsController** | `api/TeamStats` | Статистика по турниру и по команде |

**Дополнительно в pipeline:**

- **`GlobalExceptionHandler`** — единообразные JSON-ответы при ошибках.
- Swagger UI по префиксу `swagger`.
- **Статические файлы** и **`MapFallbackToFile("index.html")`** — отдача SPA; это не часть бизнес-логики API, а способ **задеплоить UI и API** из одного хоста.

## Infrastructure

- **`AppDbContext`** — маппинг сущностей: **`jsonb`** для правил и множеств id (сравнение через `ValueComparer`), **`bytea`** для бинарных фото, **complex types** для гражданства и номера у игрока.
- **Реализации репозиториев** в `SportsStats.Infrastructure.Persistence.Repositories` по интерфейсам из **Domain** (`ITournamentRepository`, `IMatchRepository`, `IPlayerRepository`, `ITeamRepository`, `ITeamStatsRepository`).
- **Миграции EF** — в `SportsStats.Infrastructure/Migrations` (эволюция схемы: матчи, правила, составы, расписание, Nullable поля и т.д.).

Строка подключения: **`ConnectionStrings:DefaultConnection`** (шаблон — `SportsStats.API/appsettings.template.json`).

## Сборка, запуск, Docker
### 1. Клонируйте репозиторий
```bash
git clone https://github.com/kulich1708/SportsStats.git
cd SportsStats
```
Далее есть 3 варианта как запустить проект
### 1. Локально:

```bash
# 1. Настраиваем строку подключения
cd SportsStats.API
# Скопировать файл appsettings.json
cp appsettings.template.json appsettings.json
# Отредактируйте, указав ваши данные
cd ../
# 2. Собираем проект
dotnet restore
dotnet build

# 3. Применяем миграции (БД уже должна быть запущена)
dotnet ef database update --project SportsStats.Infrastructure --startup-project SportsStats.API

# 4. Запускаем API
dotnet run --project SportsStats.API
```
URL по умолчанию см. `SportsStats.API/Properties/launchSettings.json` (например, `http://localhost:5121`).

### 2. Docker

**Docker Compose** поднимает API и Postgres; переменные окружения для БД и строки подключения задаются в `.env`.

``` bash
cp .env.template .env
# Отредактируйте файл .env, указав ваши данные

# Поднимите БД
docker-compose up -d db
sleep 5
docker-compose up -d --build api
# Примините миграции
docker exec sports-stats-api dotnet ef database update
# Запустите приложение
docker-compose up -d sports-stats-api
```
URL по умолчанию `http://localhost:8080/`.

### 3. Docker с восстановлением бэкапа бд
- Скачайте бэкап [ссылка](https://disk.yandex.ru/d/56iM-J7FUmFdng)
- Положите файл backup.sql в корень проекта

- Восстановите данные
```bash
# Поднять только базу данных
docker-compose up -d db

# Загрузить переменные из .env
$env:POSTGRES_USER = (Get-Content .env | Where-Object { $_ -match '^POSTGRES_USER=' }) -replace '^POSTGRES_USER=',''
$env:POSTGRES_DB = (Get-Content .env | Where-Object { $_ -match '^POSTGRES_DB=' }) -replace '^POSTGRES_DB=',''

# Скопировать и применить бэкап
docker cp backup.sql sports-stats-db:/backup.sql
docker exec sports-stats-db psql -U ${POSTGRES_USER} -d ${POSTGRES_DB} -f /backup.sql

# Запустить приложение
docker-compose up --build
```
URL по умолчанию `http://localhost:8080/`.
